# ADR-001: src/-layout вместо плоской структуры

**Статус:** принято  
**Дата:** 2025-01-01

## Контекст

При создании Django-проекта стоит выбор: разместить код в корне репозитория (плоская структура) или вынести в отдельную директорию `src/`.

Стандартный `django-admin startproject` создаёт плоскую структуру, где пакет настроек лежит рядом с `manage.py`. Это работает для простых проектов, но с ростом кодовой базы возникают проблемы:

- Корень репозитория засоряется конфигурацией (`pyproject.toml`, `Dockerfile`, `Makefile`, `.env`) и Python-пакетами одновременно.
- Сложно отличить исходный код от инфраструктурных файлов.
- `ruff`, `mypy` и другие инструменты приходится настраивать исключать конфигурационные файлы.

## Решение

Весь Python-код размещён в `src/`:

```
├── manage.py
├── pyproject.toml
└── src/
    ├── config/     ← settings, urls, wsgi, asgi
    ├── core/       ← инфраструктура
    ├── apps/       ← Django-приложения
    ├── templates/
    ├── static/
    └── media/
```

`manage.py` добавляет `src/` в `sys.path`:

```python
sys.path.append(str(Path(__file__).resolve().parent / "src"))
```

Это позволяет использовать чистые импорты без префикса `src.`:

```python
from core.models import BaseModel
from apps.users.models import User
from config.settings import DEBUG
```

Ruff настроен на `src = ["src"]` в `pyproject.toml`, чтобы корректно распознавать first-party импорты.

## Альтернативы

### Плоская структура (стандартная Django)

```
├── manage.py
├── myproject/    ← settings
├── users/        ← приложение
├── Dockerfile
└── pyproject.toml
```

**Плюсы:** привычна разработчикам Django, не нужна манипуляция с `sys.path`.  
**Минусы:** исходный код смешан с конфигурацией, нет чёткой границы «что деплоить».

### Monorepo с packages/

**Плюсы:** масштабируется на несколько сервисов.  
**Минусы:** избыточно для одного Django-проекта.

## Последствия

**Положительные:**
- Чистое разделение: корень = конфигурация, `src/` = код.
- Инструменты (`ruff`, `mypy`) направлены на `src/` — меньше шума.
- Gunicorn запускается с `--chdir src`, работает с чистыми импортами.
- Dockerfile копирует `src/` как единый артефакт.

**Отрицательные:**
- `manage.py` содержит дополнительную строку (`sys.path.append`).
- Новому разработчику нужно знать, что импорты пишутся без `src.`.
- `make startapp` требует патчинг `apps.py` (автоматизировано в Makefile).
